== Entwurfsdokumentation

=== Annahmen und Abhängigkeiten

==== Annahmen
* Nutzer verfügen über eine stabile Internetverbindung.
* Nutzer verwenden ein Endgerät mit HTML5-fähigem Browser und aktiviertem JavaScript.
* Gültige SSL-Zertifikate sind vorhanden und werden vom Browser akzeptiert.

==== Abhängigkeiten
* Die Datenbank wird extern über Supabase gehostet.
* Backend und Frontend werden lokal in Docker-Containern betrieben.
* Internetzugang wird ausschließlich zum Herunterladen des Docker-Images sowie zur Kommunikation mit der Datenbank benötigt.

=== Architektur-relevante Anforderungen

==== Begriffserklärung
* *FA* – Functional Requirements
* *NFAU* – Non-Functional Application Usability
* *NFAS* – Non-Functional Application Supportability
* *NFAP* – Non-Functional Application Performance
* *NFAR* – Non-Functional Application Reliability

==== Funktionale Anforderungen
* FA-1: Persistente Speicherung von Benutzerdaten.
* FA-2: Zugriffsbeschränkung für authentifizierte Nutzer.
+
Hinweis: Es existiert aktuell kein separater Administratorzugriff. Änderungen an Daten erfolgen manuell direkt in der Datenbank.

==== Benutzbarkeit
* NFAU-1: Das System ist intuitiv bedienbar.
* NFAU-2: Das Erfassen einer Einnahme oder Ausgabe dauert maximal 60 Sekunden.

==== Qualitätsanforderungen

===== Wartbarkeit
* NFAS-1: Modularer Aufbau zur Erleichterung von Wartung und Erweiterung.
* NFAS-2: Umfassende und verständliche Dokumentation.

===== Performance
* NFAP-1: Schneller Seitenaufbau und kurze Reaktionszeiten.
* NFAP-2: Unterstützung von mindestens 50 gleichzeitigen Nutzern.
* NFAP-3: Effiziente Datenbankabfragen.
* NFAP-4: Die Anwendung ist innerhalb weniger Minuten startbar.
+
Ein kontrollierter Herunterfahrmechanismus ist nicht explizit implementiert; Prozesse können jederzeit beendet werden.

===== Zuverlässigkeit
* NFAR-1: Dialogfelder reagieren ohne wahrnehmbare Verzögerung.
* NFAR-2: Validierung verhindert fehlerhafte Benutzereingaben.
* NFAR-3: Maximale Wartungszeit von 1 % der Gesamtbetriebszeit.
* NFAR-4: Regelmäßige Backups der Datenbank erfolgen durch den Datenbankanbieter.

=== Architekturkonzept und Entscheidungen

==== Architekturansatz
Das System basiert auf einer schichtenbasierten Webarchitektur.
Frontend und Backend sind strikt getrennt und kommunizieren über eine REST-Schnittstelle.
Die Anwendung wird als Single-Page-Application mit Nuxt 3 umgesetzt.

==== Begründung
* Klare Trennung von Benutzeroberfläche und Datenzugriff erhöht Wartbarkeit.
* Komponentenbasierte Entwicklung erleichtert Erweiterungen.
* Nutzung etablierter Technologien mit aktiver Community.

=== Systemarchitektur

==== Logische Sicht
Backend::
Controller → Service → Domain → Repository
Frontend::
Nuxt 3 (Pages, Components, Composables)
Datenbank::
Supabase (PostgreSQL)

==== Physische Sicht
Frontend und Backend laufen lokal in Docker-Containern.
Die Datenbank wird extern über Supabase betrieben.
