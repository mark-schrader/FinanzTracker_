== Praktiken und Bewertung

=== Verwendete Management- und technische Praktiken im Projekt

In unserem Projekt verwenden wir eine angepasste Form des *agilen Scrum-Ansatzes*, die auf unsere verfügbaren Kapazitäten (ca. 5 Stunden pro Woche pro Teammitglied, inkl. 1,5 Stunden Teammeeting alle zwei Wochen) zugeschnitten ist. Die wichtigsten management- und technischen Praktiken, sowie die Fortschrittsverfolgung sind wie folgt organisiert:

==== Iterative Entwicklung (Scrum-Sprints)

* Im letzten Semester haben wir in *einwöchigen Sprints* gearbeitet, die in 1,5-stündigen Meeting geplant, überprüft und reflektiert wurden.
* Jede Woche wurden wenige, priorisierte Aufgaben (z. B. 1–2 Story Points pro Mitglied) umgesetzt.

* Wir haben unsere Arbeitsstrategie im 2. Semester abgeändert. Es finden *einwöchige Meetings* innerhalb des Dev-Teams und des Orga-Teams statt und *zweiwöchige Meetings* mit dem gesamten Team statt. In den großen Meetings wird der Plan für den nächsten Sprint besprochen und in den kleinen Meetings dann die Umsetztung in den zwei Wochen bis zum nächsten Woche.

==== Continuous Integration (CI)

* Code wird regelmäßig in ein gemeinsames Repository gepusht.
* Eine automatisierte CI-Pipeline (GitHub) führt bei jedem Commit Builds und Tests aus.
* Änderungen werden kleinteilig entwickelt, um Merge-Konflikte und Integrationsprobleme zu vermeiden.
* Bei der Arbeit mit verschiedenen Branches wird immer ein Pull-Request erstellt, wenn ein persönlicher Branch in den Main-Branch gemergt werden soll. Die Entscheidung über die Annahme des Pull-Requests erfolgt nach dem Vier-Augen-Prinzip. Das bedeutet, dass neben dem/der Entwickler*in eine weitere Person aus dem Team oder unser Coach den Code auf Richtigkeit und Qualität überprüft.

==== Unabhängiges Testen

* Testaufgaben werden möglichst von Teammitgliedern übernommen, die nicht direkt am betroffenen Feature gearbeitet haben.
* Es werden einfache manuelle Tests sowie automatisierte Unit-Tests durchgeführt. (evtl. noch ueberarbeiten)
* Fehler werden dokumentiert und im nächsten Sprint behoben.

=== Definition of Ready
// Listen Sie hier die Kriterien auf, die ein Product Backlog Item (z.B. User Story) erfüllen muss, damit es bei der Sprint-Planung berücksichtigt werden kann
Die Definition of Ready legt fest, wann eine User Story oder ein Task so weit vorbereitet ist, dass er im nächsten Sprint umgesetzt werden kann. Das bedeutet, dass alle notwendigen Informationen und Anforderungen geklärt sind und das Dev-Team genau versteht, was umgesetzt werden soll.

=== Definition of Done
// Listen Sie hier alle Kritierien auf, damit die Aufgabe eines bestimmten Typs (z.B. Task, User Story) als abgeschlossen gilt.
Die Definition of Done beschreibt, wann eine User Story oder ein Task als vollständig abgeschlossen gilt. Dafür müssen alle Akzeptanzkriterien erfüllt sein, damit die Aufgabe als erledigt gilt.
